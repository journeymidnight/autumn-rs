Task scope: F109 — physical extent file deletion when refs → 0
(`punch_holes` / `truncate` did not unlink replica `.dat`/`.meta`
files; deferred from the F108 session as "Issue 2").

Root cause (verified by code-reading + Go reference + reproduced in
new integration test):

  When `handle_stream_punch_holes` or `handle_truncate` decremented
  an extent's `refs` to 0, the manager removed the
  `MgrExtentInfo` from the in-memory store and deleted the
  `extents/{id}` etcd key via `mirror_stream_extent_mutation`. It
  did NOT notify any extent-node to unlink the physical
  `extent-{id}.dat` + `.meta` files.  Result: every replica's data
  dir kept the bytes forever after `autumn-client gc`. The user hit
  this concretely after the F105/F106/F107/F108 session — manager
  freed metadata, `du -sh /tmp/autumn-rs/d*/` stayed flat.

  Go reference (`/Users/zhangdongmao/upstream/autumn`) solves this
  via etcd-watch on every node:
  `node/smclient/extent_info_manager.go:140-158` watches `extents/`,
  on DELETE event calls `node.go:138 RemoveExtent → diskfs.go:108
  os.Remove(extentName)`. No fanout RPC; each replica converges
  independently from etcd state.

  The Rust port has diverged: extent-nodes have NO etcd client at
  all; every manager↔node command (`require_recovery`, `re_avali`,
  `copy_extent`, `df`, …) flows over autumn-rpc. Adding etcd watch
  to extent-node would require extending `autumn-etcd` from
  single-key to prefix watch and shipping a new client surface —
  much larger blast radius. Per user decision (recorded after the
  brainstorming round), F109 takes the manager-push path —
  matching the existing Rust pattern — and keeps extent-node
  etcd-free.

Fix (single commit):

  1. Wire codec — `crates/stream/src/extent_rpc.rs`:
     - New const `MSG_DELETE_EXTENT: u8 = 11`.
     - New rkyv `DeleteExtentReq { extent_id: u64 }`. Response
       reuses the existing `CodeResp`.
     - Round-trip unit tests (`tests` mod, two cases).

  2. Manager-side wire types — `crates/rpc/src/manager_rpc.rs`:
     - New const `MSG_RECONCILE_EXTENTS: u8 = 0x31` (StreamManager
       service slot, next free in 0x30 range).
     - New const `EXT_MSG_DELETE_EXTENT: u8 = 11` (manager → node
       call).
     - New rkyv types `ReconcileExtentsReq`,
       `ReconcileExtentsResp`, `ExtDeleteExtentReq`.

  3. Extent-node delete handler —
     `crates/stream/src/extent_node.rs`:
     - `MSG_DELETE_EXTENT` route in `dispatch`.
     - New `handle_delete_extent`: F099-M owner-shard forward, then
       `extents.remove(&id)` (so subsequent appends fast-fail with
       NotFound), then `DiskFS::remove_extent_files` to unlink
       `.dat` + `.meta`. Idempotent — `NotFound` errors on either
       file are downgraded to `Ok`. Returns `CodeResp { CODE_OK |
       CODE_ERROR }`.
     - New `DiskFS::remove_extent_files` helper (uses existing
       `extent_path` / `meta_path`).

  4. Extent-node startup reconcile —
     `crates/stream/src/extent_node.rs`:
     - `ExtentNode::new` calls
       `reconcile_orphans_with_manager()` after `load_extents()`.
       Sends every locally loaded `extent_id` (filtered by
       `owns_extent` for F099-M) via `MSG_RECONCILE_EXTENTS` to the
       manager; receives back the subset that's no longer in
       `s.extents`; unlinks the corresponding files. Best-effort:
       a network failure logs at WARN and doesn't block startup.

  5. Manager pending queue + dispatch loop —
     `crates/manager/src/extent_delete.rs` (new module):
     - `PendingDelete { extent_id, pending_addrs: Vec<String>,
       attempts: u32 }`.
     - `pending_extent_deletes: Rc<RefCell<VecDeque<PendingDelete>>>`
       field added to `AutumnManager`; init in `new()`.
     - `extent_delete_loop`: sweep every 2 s, drain pending,
       fan out `EXT_MSG_DELETE_EXTENT` via `ConnPool` per replica.
       Drop confirmed addrs from each entry. After 60 failed
       sweeps (≈ 2 min) drop the entry and log WARN — orphans
       reaped on next node-startup reconcile.
     - `Self::snapshot_replica_addrs(&nodes, eid, extent)` helper
       resolves `replicates ++ parity` node-ids to shard-routed
       addresses via the existing `Self::shard_addr_for_extent`.
     - Spawn from `start_runtime_tasks`.

  6. Snapshot wiring in handlers —
     `crates/manager/src/rpc_handlers.rs`:
     - `handle_stream_punch_holes` (lines ≈793-887) and
       `handle_truncate` (lines ≈901-1002): replaced
       `let mut s = self.store.inner.borrow_mut();` with
       `let mut guard = ...; let s: &mut MetadataState =
       &mut guard;` so partial-borrows on disjoint fields work
       (`RefMut` deref alone confuses NLL).
     - For each `extent_id` with `extent.refs <= 1`: snapshot the
       replica addresses via `snapshot_replica_addrs` BEFORE the
       in-memory `s.extents.remove(&extent_id)`. Implementation
       uses an intermediate `HashMap<u64, MgrExtentInfo>` snapshot
       to dodge the nested-borrow issue cleanly.
     - After `mirror_stream_extent_mutation` returns OK, call
       `self.enqueue_pending_deletes(pending_deletes)` —
       etcd-first ordering preserved.
     - New `handle_reconcile_extents`: `ensure_leader`, decode
       `ReconcileExtentsReq`, return the subset of `req.extent_ids`
       not in `s.extents`. Read-only.
     - `MSG_RECONCILE_EXTENTS` route added to `dispatch`.

  7. Test infrastructure — `crates/stream/tests/test_helpers.rs`:
     - Added `TestConn::delete_extent` for `f109_extent_delete`.

  8. New integration tests:
     - `crates/stream/tests/f109_extent_delete.rs` (3 tests):
       existing extent unlinks .dat+.meta; missing extent is
       idempotent; delete-then-realloc starts fresh.
     - `crates/manager/tests/f109_physical_deletion.rs`: end-to-end
       — drives `StreamClient::punch_holes` with `max_extent_size =
       1024` to roll 8 extents, punches 6 of them, polls until
       both replicas have unlinked the punched extents within 8 s,
       asserts the surviving extents remain on at least one replica.
     - `crates/manager/tests/f109_offline_reconcile.rs`: writes a
       synthetic orphan `extent-{id}.dat`+`.meta` into the correct
       hash subdir of a tempdir BEFORE `ExtentNode::new`, then
       starts the node pointing at that dir with the manager
       endpoint configured. Verifies the startup reconcile path
       unlinks the orphan within 4 s.
