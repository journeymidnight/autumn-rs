TaskStatus: completed
Task scope: gallery /get/{name} Range handler — stream the response in
4 MiB chunks instead of materialising the entire `bytes=N-` slice as a
single Vec. Fix `bytes=-N` (suffix range) parsing along the way.

Current summary:
  Symptom: `ffmpeg -i http://127.0.0.1:5001/get/<file>.mp4 -c copy ... HLS`
  was very slow, and the bigger the source file the worse it got.

  Root cause: ffmpeg's HTTP demuxer issues `Range: bytes=N-`. The previous
  handler computed `length = total - N` and called `get_range(N, length)`
  in one shot — for a 1 GiB upload that allocated a 1 GiB Vec (and pulled
  a 1 GiB autumn-rpc payload back to the gallery process) before any HTTP
  body byte shipped. ffmpeg blocked on the entire buffer materialising.
  The autumn-rs storage layer's `get_range` is already a true partial
  read end-to-end; the bottleneck was purely in the gallery response
  shape.

  Fix:
    - New `parse_byte_range(s, total)` helper in
      examples/gallery/src/main.rs handles `bytes=N-`, `bytes=N-M`, and
      `bytes=-N` (suffix). The previous code mis-parsed `bytes=-N` as
      `bytes=0-N`.
    - New `stream_kv_range(client, metrics, name, start, end_inclusive)`
      helper builds a `Body::from_stream` driven by an mpsc producer that
      loops `get_range(off, RANGE_CHUNK_BYTES = 4 MiB)` until done. The
      producer task is `SendWrapper`'d (Rc/RefCell client) and detached on
      compio; chunks travel as `Bytes` over the (Send) mpsc into axum.
      Backpressure cap=2 keeps memory at ~8 MiB regardless of value size.
    - `get_handler_inner` is rewritten to: head() once → parse Range →
      build streaming Body → 200 or 206 with the right Content-Length /
      Content-Range. The non-Range path streams uniformly too.
    - The same per-chunk loop releases the `Rc<Mutex<ClusterClient>>`
      between fetches, so other gallery handlers can interleave instead
      of waiting on one giant get_range.
