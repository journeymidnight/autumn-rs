Task scope: F108 — manager `EtcdClient` panics with `RefCell already
borrowed` under concurrent RPCs (hit live by the user as the 4-partition
GC race against shared extent 10).

Root cause (verified by code-reading + Go reference + reproduced in test):

  `crates/etcd/src/lib.rs::EtcdClient::unary_call` line 200 / line 216,
  and `LeaseKeeper::keep_alive` line 273 / line 283, all wrote
  `self.channel.borrow_mut().call(path, body).await`. The
  `RefMut<GrpcChannel>` guard from `borrow_mut()` was held across the
  `.await`. On the manager's single-threaded compio runtime, when a
  second task entered `unary_call` (or any of `put` / `get` / `txn` /
  `delete` / `lease_grant` / `lease_revoke` — all funnel through
  `unary_call`) while the first was still awaiting the response,
  `borrow_mut()` panicked.

  User-reported scenario: triggering `autumn-client gc` for 4 partitions
  in quick succession (parts 26 / 32 / 20 / 15 — all sharing extent 10
  from the F105/F106/F107 session) caused 4 concurrent
  `handle_stream_punch_holes` invocations on the manager. Each handler
  awaits `mirror_stream_extent_mutation → EtcdClient::txn`, and the
  second one panicked with `already borrowed: BorrowMutError`. User
  workaround was "GC one partition at a time"; F108 removes that.

  This was already a known clippy warning (FCI-01: "RefCell reference
  held across await point (4 in autumn-etcd)") — F108 is the runtime
  manifestation.

Fix (single commit):

  • `crates/etcd/src/transport.rs`:
    - `GrpcChannel::call(&mut self, ...)` removed.
    - New `GrpcChannel::sender(&self) -> http2::SendRequest<Full<Bytes>>`
      returns a cheap clone of the underlying hyper sender (mpsc handle
      + Arc — hyper's canonical idiom for HTTP/2 request multiplexing).
    - New free function `call_with_sender(sender, path, body)` does the
      actual gRPC POST + response-decode work. Free function (not
      method) so callers can clone-then-drop-borrow before await.

  • `crates/etcd/src/lib.rs`:
    - `EtcdClient::unary_call`: `let mut sender = self.channel.borrow().sender();`
      (RefCell guard dropped at `;`), then
      `call_with_sender(&mut sender, path, body).await`. Both first
      attempt and post-reconnect retry.
    - `LeaseKeeper::keep_alive`: same pattern.
    - `reconnect_shared` is unchanged — its `*channel.borrow_mut() =
      new_channel` is synchronous (the await is on the prior
      `GrpcChannel::connect` call), so no borrow-across-await there.

  • `crates/etcd/CLAUDE.md`:
    - "Threading Model" section adds the F108 invariant: clone the
      sender out of the RefCell before await, never hold `RefMut`
      across `.await`. Includes the rule for future RPC additions.

  • `crates/etcd/tests/concurrent_calls.rs` (new):
    - `concurrent_puts_no_borrow_panic` — 8 concurrent `put` tasks on
      one shared `Rc<EtcdClient>`; pre-F108 panics with
      `RefCell already borrowed` at lib.rs:200, post-F108 all green.
    - `concurrent_mixed_no_borrow_panic` — 4 puts + 4 gets concurrently.
    - `concurrent_stress_no_borrow_panic` — 16 tasks, 1ms warmup sleep
      to maximise interleaving.
    - All tests skip cleanly if etcd is not at 127.0.0.1:2379.

  • `feature_list.md`:
    - F108 entry added with target / evidence / verification /
      acceptance / notes; `passes: true` after the empirical
      bug-reproduction round-trip below.
