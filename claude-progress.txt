Task scope: F115 — fix EINVAL on >2 GiB extents in local-file I/O paths + surface EC status in `autumn-client info`

Current summary:
  Two related fixes:

  Part A — `autumn-client info` EC visibility:
    `InfoExtentView` gains `parity: Vec<u64>` field. Text rendering prints
    `EC(K+M), data=[...], parity=[...]` when parity is non-empty; otherwise
    `replicas=[...]`. JSON projection exposes the same field. Rule:
    `!parity.is_empty()` ⇔ extent is currently EC-encoded.

  Part B — F115: chunked local-file I/O on ExtentNode:
    F105 (commit d80a2b8) chunked `StreamClient::read_bytes_from_extent` at
    the RPC path. Six extent_node.rs call sites still issued single oversize
    syscalls that returned EINVAL on macOS (INT_MAX ≈ 2 GiB) and Linux
    (0x7ffff000) for extents > 2 GiB. Symptoms: ec_conversion_dispatch_loop
    (every 5 s) spamming "Invalid argument (os error 22)" for extents 33, 42,
    46 (each ~3 GiB).

    Added `file_pread_chunked` and `file_pwrite_chunked` helpers
    (FILE_IO_CHUNK_BYTES = 256 MiB) to extent_node.rs (~line 443-492).
    Both fast-path the common case (single syscall when <= 256 MiB).
    `file_pwrite_chunked` uses `Bytes::split_to` (O(1), no copy).

    Six call sites updated:
      handle_convert_to_ec  -> file_pread_chunked
      handle_read_bytes     -> file_pread_chunked
      handle_copy_extent    -> file_pread_chunked
      run_recovery_task     -> file_pwrite_chunked
      handle_re_avali       -> file_pwrite_chunked
      write_shard_local     -> file_pwrite_chunked

  Verification (live cluster):
    After releasing the fixed binary (cluster.sh restart-node 1-4):
    - extent 33 (3.0 GB): EC(3+1), data=[1, 3, 5], parity=[7], eversion=3 [OK]
    - extent 42 (3.0 GB): EC(3+1), data=[1, 3, 7], parity=[5], eversion=3 [OK]
    - extent 46 (3.1 GB): EC(3+1), data=[1, 3, 5], parity=[7], eversion=3 [OK]
    - No more EINVAL errors in manager log after restart
    - Physical files on all 4 nodes: ~1 GiB each (shard-sized, down from ~3 GiB)
