TaskStatus: completed
Task scope: F103 — fix "$AC split N 一直block" bug. After a partition has
been split once, a second `autumn-client split <PARTID>` would hang ~25s
and return an opaque "rpc status FailedPrecondition: multi_modify_split
failed: precondition failed: mid_key is not in partition range" error.

Root cause (verified by inspection + live repro):
  PartitionServer::sync_regions_once skips the rg refresh for an already-
  open partition (lib.rs:884 `if self.partitions.borrow().contains_key
  (&part_id) { continue; }`). After the 1st split the manager has the new
  narrowed range, but the PS-local PartitionData.rg is the pre-split wide
  range. Consequences cascade:
    1. open_partition's overlap detection ran against the wide rg, so
       has_overlap stayed 0 → F037's "cannot split: has overlapping keys"
       gate (handle_split_part:304) is silently bypassed.
    2. unique_user_keys() iterates CoW-shared SSTables that still span
       the original wider range, so the returned key list crosses the
       manager-authoritative mid_key boundary.
    3. handle_split_part picks mid = sorted[len/2] against this
       unfiltered set; it frequently lands above the manager's narrowed
       end_key.
    4. multi_modify_split rejects ("mid_key is not in partition range").
    5. The in-handler retry loop backs off 100→200→400→800→1600→2000×
       2000×2000 ms = 9.1s × 2 ClusterClient.call_ps_for_part retries
       ≈ 25 s wallclock for a deterministic precondition failure.

Fix (crates/partition-server/src/rpc_handlers.rs::handle_split_part):
  (a) Fetch authoritative rg from the manager via MSG_GET_REGIONS before
      picking mid_key. Use the partition's per-PS pool + manager_addr
      (previously _-prefixed unused params).
  (b) Filter unique_user_keys() to keys in the authoritative rg. If
      <2 in-range keys, return FailedPrecondition with "run major
      compaction first" instead of letting the manager reject mid_key.
  (c) On successful multi_modify_split, narrow PS-local part.rg to
      `[auth_rg.start, mid)` AND re-evaluate has_overlap against the
      current sst_readers (smallest_key/biggest_key vs the new range).
      Without (c) the same bug recurs on the 3rd split.

Verification (live, fresh 4-replica cluster, post-fix release binary):
  ✓ put 20 small keys → split 15 → succeeds (partitions 15 + 19)
  ✓ immediate `split 15` again → returns INSTANTLY with
    `cannot split: partition has overlapping keys` (was ~25 s pre-fix)

Outstanding / deferred:
  - Architecturally cleaner fix would be to make sync_regions_once
    propagate rg changes to existing partitions and re-evaluate
    has_overlap on rg change. That requires cross-thread (main →
    P-log) state propagation because PartitionData is Rc<RefCell> /
    !Send. Deferred — current scoped fix is self-contained on the
    partition thread that already owns PartitionData.
  - User additionally reported "在写数据的时候split，然后重启失败"
    accompanied by a `partition thread failed: open SST extent=21
    offset=0 read_len=24417 preview=[00, 00, 3f, 00, 50, 61, 74, ...]`
    error from the pre-fix cluster's stored data. The 32-byte preview
    is a valid SSTable entry header (overlap=0, diff_len=63, key=
    "Patreon--leeesovely-Apr-2023..."), so the parse failure is in the
    SSTable's meta-block trailer (last 4 bytes = meta_len, then the
    MetaBlock with magic AU7B). We could not reproduce on the post-fix
    binary across several aggressive concurrent-write + split +
    kill -9 + restart cycles. The original on-disk extent file was
    wiped by `cluster.sh clean` before we could inspect it. If this
    recurs, capture `/tmp/autumn-rs/d{1..4}/<hash>/extent-21.dat`
    BEFORE running cluster.sh clean so we can decode the bytes and the
    meta_stream checkpoint that referenced them.
