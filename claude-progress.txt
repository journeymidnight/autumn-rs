Current summary:
  Field evidence now shows this is not a gallery/list bug:
  1. original keys such as `Patreon--...-300.jpg` are `head/get` misses
  2. matching thumbs still exist and are queryable
  3. offline checkpoint inspection shows the current live SST/meta view misses
     the original while still hitting the thumb
  4. raw bytes for the original key still exist in older row extents on disk
  5. direct full-SST dump of the live part-40 table (extent 41) confirms this is
    not a point-lookup/seek bug: the table physically contains
    `.thumb/...241.jpg` through `.thumb/...301.jpg`, then only
    `Patreon--...-003.jpg`; `Patreon--...-300.jpg` is absent from the live SST

  Additional split-stream finding:
  6. `StreamClient::commit_length()` / manager `check_commit_length` return the
    commit length of the tail extent only, not a whole-stream logical length;
    however `compute_duplicate_stream()` clones all extents and does not use the
    sealed-length argument to crop historical row extents, so tail-length
    semantics do not currently explain the missing originals.
    7. old shared row extent 12 contains a valid SST slice at `offset=0 len=55706`
      whose range is `.thumb/...001.jpg` .. `Patreon--...301.jpg`, and normal
      SST lookup on that slice hits `Patreon--...300.jpg`
    8. a focused system test with exactly one `split -> compact(right child)`
      passes, so the field bug is not reproduced by the minimal single-split path
      9. root cause for the focused two-round failure was not overlap logic itself
        but sealed-tail handling on duplicated row streams: after split, the new
        child stream could start with a manager-sealed tail extent, but
        `StreamClient::ensure_tail_initialised()` still seeded the worker with that
        sealed tail instead of allocating a fresh extent
      10. that left descendant compaction appending against the duplicated tail;
        before the fix this surfaced as `LockedByOther: a newer owner holds extent 8`
        and overlap never cleared, blocking the next split
      11. fixing sealed-tail initialisation to allocate a fresh extent immediately
        makes the focused two-round path pass, and the broader three-round
        gallery regression now passes as well

  Working hypothesis: repeated split + overlap-clearing compaction on a
  gallery-like mixed keyspace (small `.thumb/...` keys + large original image
  keys) can drop large original keys from the rightmost child's live SST/meta
  lineage.

  2. test writes large original values plus small thumb keys
  3. test repeatedly splits and compacts the rightmost child
  4. after each round it asserts a high original key and its thumb still exist
