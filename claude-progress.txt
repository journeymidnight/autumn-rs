TaskStatus: completed
Task scope: F121 — node-failure write recovery. Two reported symptoms after
`cluster.sh stop-node 1` against a 4-node cluster whose tail extents were
on replicas [1, 3, 5]:
  (a) `autumn-client put k2` blocks forever (>120 s observed).
  (b) `autumn-client info` keeps reporting the dead node's disk
      `online=true` indefinitely.

Root cause:
  After SIGTERM kills extent-node 1, the kernel sends FIN to the PS-side
  TCP socket. autumn-rpc's `read_loop` sees EOF, returns `Ok(())`, and
  clears `pending`. But the `Rc<RpcClient>` is *still in
  crates/stream/src/conn_pool.rs's `ConnPool`* — the pool only ever evicts
  on `call*`-error, and `pool.send_vectored` did not evict on error at
  all. The next `pool.send_vectored("127.0.0.1:9101", …)` returns the
  same dead client. `client.send_vectored` happily inserts a fresh
  `(req_id → oneshot::Sender)` into `pending` (no read_loop alive to
  dispatch it), succeeds at submitting (writer_task still drains the
  channel; macOS loopback writes to half-closed sockets succeed for
  small payloads), and the caller's `rx.await` hangs forever. There was
  no transport-layer timeout on the append fanout (Go autumn has 5 s
  via `context.WithTimeout` — `streamclient.go:770`). The manager's
  `select_nodes` did not consult disk liveness, so even after the
  retry path fired the manager picked node 1 first. And
  `disk_status_update_loop` only did *positive* updates — `Err(_) =>
  continue` left a stale `online=true`. A fourth pre-existing bug
  finally surfaced: the per-disk-id status update path keyed on the
  extent-node's local `disk_id` (e.g. `--disk-id 4`) which never
  matches the manager-allocated `disk_id` (e.g. 8), so even a
  successful df was a silent no-op against `s.disks`.

Fix (this session — five layered changes, all small):
  Files changed:
    - crates/rpc/src/client.rs — add `closed: Rc<Cell<bool>>` to
      RpcClient; set true on read_loop / writer_task exit; short-
      circuit `send_frame` / `send_vectored` / `send_oneshot` with
      `RpcError::ConnectionClosed`. New `is_closed()` accessor. New
      unit test `closed_flag_set_after_peer_disconnect`.
    - crates/stream/src/conn_pool.rs — `get_client` evicts cached
      entries with `is_closed() == true`; `send_vectored` evicts on
      submit error when `client.is_closed()` matches the existing
      `call*` semantics.
    - crates/stream/src/client.rs — new `append_fanout_timeout`
      (env `AUTUMN_STREAM_APPEND_TIMEOUT_MS`, default 5 s, clamped to
      [200 ms, 60 s]). `launch_append` wraps each replica's response
      receiver in `compio::time::sleep + futures::future::select`;
      `Elapsed` becomes a soft error that the existing
      `append_payload_segments` retry loop escalates to
      `alloc_new_extent`.
    - crates/manager/src/recovery.rs — `disk_status_update_loop` and
      `recovery_collect_loop`'s df poll now call
      `mark_node_disks_offline(store, node)` on RPC error and
      `mark_node_disks_online(store, node)` on success; both key on
      `MgrNodeInfo.disks` (manager-allocated disk_ids), sidestepping
      the long-standing local-vs-allocated disk_id mismatch in the
      df response payload.
    - crates/manager/src/lib.rs — `select_nodes` now takes
      `&HashMap<u64, MgrDiskInfo>`; pre-filters nodes whose
      `node.disks` has any `s.disks[id].online == true`. Falls back
      to the full node set when too few healthy candidates appear
      (preserves the existing fall-back path in
      `handle_stream_alloc_extent`).
    - crates/manager/src/rpc_handlers.rs — pass `&s.disks` to the
      two `Self::select_nodes` callers (`handle_create_stream`,
      `handle_stream_alloc_extent`).
    - feature_list.md — F121 entry inserted between F120 and F116.

Reference patterns leaned on:
  - Go autumn streamclient/streamclient.go:770 (5 s
    `context.WithTimeout` on append fanout) → Layer 3.
  - Go autumn streamclient/streamclient.go:917-941 (any error → 2×
    local retry → `MustAllocNewExtent`) → autumn-rs already had this
    in `append_payload_segments`; F121 just unblocks the trigger.
  - Go autumn manager/stream_manager/policy.go:16-46
    (`AllocExtent` filters by `getAllNodeStatus(true)`) → Layer 5.
  - F119 typed-signal pattern (server returns
    `CODE_EVERSION_MISMATCH` → client invalidates cache + retries)
    → Layer 1 reuses the same shape one layer down: the closed flag
    is the typed signal, the conn-pool evict is the
    cache-invalidate.

Key invariants preserved:
  - autumn-rpc single-threaded compio: `closed.get()` and
    `pending.borrow_mut().insert(...)` happen in the same sync block
    (no awaits between them), so the close-vs-send race is bounded
    to "cleared after we inserted" → caller's `rx.await` returns
    `Err` immediately via the existing `pending.clear()` path.
  - `MgrNodeInfo.disks: Vec<u64>` already carries the manager's
    allocated disk_ids; F121's `mark_node_disks_*` just consume that
    list, no schema change.
  - `select_nodes` falls through to the full sorted set when too few
    online disks are visible (cold leader before the first df sweep
    has run) — `handle_stream_alloc_extent`'s per-RPC fall-back is
    still the load-bearing layer here, F121 just narrows the
    candidate set in the common case.
