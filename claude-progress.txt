Task scope: F105 + F106 + F107 — three-fix bundle to unblock GC on the
user's running 4-partition cluster (extent 10 = 3.0 GiB CoW-shared

Root cause (verified by code-reading + Go reference + live PS log):

  F105 — `StreamClient::read_bytes_from_extent` issues one `pread(file,
    offset, length)` syscall per RPC. macOS caps `pread` at INT_MAX
    (~2 GiB); Linux at 0x7ffff000. The user's GC tries to read 3.0 GiB
    in one shot → server-side EINVAL. Same code path is used by
    `recover_partition` (lib.rs:2381) — recovery would fail to open
    any partition with a >2 GiB sealed extent on the next PS restart.
    "Ticking time bomb": currently extent 42 is open at 2.9 GiB; once
    it seals it'll join extent 10 in the unrecoverable bucket.

  F106 — `run_gc` calls `read_bytes_from_extent(eid, 0, sealed_length)`
    and decodes the entire returned Vec into records in memory. Peak
    RAM ≈ sealed_length per partition. Also held `RefMut<PartitionData>`
    across `part_sc.append.await` per record — would panic if any
    other P-log task tried to borrow `part` during the in-flight RPC.

  F107 — compact loop's early-return `if tbls.len() < 2 &&
    has_overlap == 0 { continue; }` is correct logic (matches Go
    `range_partition/compaction.go:171-174`) but silent, and
    `open_partition` never logged final state. Result: a user-issued
    `compact <PARTID>` that is gated by this branch returns "compact
    triggered" but does nothing visible. Cost in this session: ~30
    minutes of wrong-direction debugging trying to figure out why
    compaction wasn't running.

Fix (single bundled commit):

  F105 — `crates/stream/src/client.rs::read_bytes_from_extent`:
    • Resolve effective length first: sealed → `ExtentInfo.sealed_length`;
      open → new helper `commit_length_for_extent` (min over replicas).
    • If resolved > `read_chunk_bytes()` (env
      `AUTUMN_STREAM_READ_CHUNK_BYTES`, default 256 MiB), loop with
      `read_replicated_with_failover` per chunk and concat.
    • EC path (`ec_subrange_read`) untouched — per-shard reads are
      already bounded by `sealed_length / data_shards`.

  F106 — `crates/partition-server/src/background.rs::run_gc`:
    • Rewritten as streaming chunked carry-forward. Chunk size from
      env `AUTUMN_PS_GC_READ_CHUNK_BYTES` (default 64 MiB; matches
      Go's ~1000-block replayLog window).
    • Extracted `process_gc_chunk` helper that decodes complete
      records, returns consumed-byte count; caller carries the partial
      tail into the next chunk.
    • Per record: stage new log entry under a tightly-scoped
      `borrow_mut` (seq + encode), DROP the guard, await
      `part_sc.append`, then re-acquire `borrow_mut` to update vp head
      and memtable. No more borrow-across-await.
    • Final guard: non-empty carry at end-of-loop refuses to punch
      (sealed-extent records should be byte-aligned).

  F107 — observability:
    • `background_compact_loop` early-return now emits
      `tracing::info!("compact part {}: skipped — tables=N, has_overlap=0
      (...)")`.
    • `open_partition` now emits `tracing::info!("open_partition:
      ready", part_id, tables, sst_readers, has_overlap, max_seq,
      vp_extent_id, vp_offset)` just before constructing PartitionData.

Notes for future operators / ops:
  • The Bug #2 I initially flagged ("major compact silently skipped on
    split children") turned out NOT to be a bug — Go has the same
    early-return. The actual situation is: at startup all 4 partitions
    had `tables.len() < 2 && has_overlap == 0` (likely because
    metaStream's TableLocations checkpoint only includes in-range SSTs
    by the time the cluster reaches steady state). The space waste
    sits entirely in logStream sealed extents and is reclaimable by
    GC alone — no compaction needed. F107's logging makes this case
    diagnosable.
  • If a future user reports recovery failures with the same EINVAL
    message, F105 is in the relevant code path. Look for "rpc status
    Internal: Invalid argument (os error 22)" in PS logs and check
    extent sizes via `autumn-client info` — anything sealed > 2 GiB
    pre-F105 is suspect.
