Task scope: F116 — fix slow gallery /get/ + high autumn-extent CPU after EC conversion (stale extent_info_cache → eversion-based invalidation on the read path)

Current summary:
  Symptom: after `ec_conversion_dispatch_loop` flips a sealed extent
  from 3-replica to EC, the next gallery GET (port 5001) is very slow
  (multiple seconds, sometimes 9+ s) but eventually succeeds, and
  `autumn-extent` shows elevated CPU during the transition window.

  Root cause: `StreamClient.extent_info_cache` is never invalidated
  when the manager bumps the extent's `eversion` and rewrites
  `replicates`/`parity`. The PS reads the extent via the cached
  pre-EC layout, hits 3 stale-replica addresses (whose `.dat` has
  been truncated to a shard), and burns up to 3 s per replica
  on `pool.call_timeout` before falling out. The client previously
  passed `eversion: 0` in `ReadBytesReq`, which **disabled** the
  server's own eversion guard. And `write_shard_local` never bumped
  the local `entry.eversion`, so even if the client passed a real
  eversion the server's view would still match the pre-EC value.

  Fix (3 layers, all in-band, no manager push):

    1. ExtConvertToEcReq + ConvertToEcReq + WriteShardReq each gain
       a `eversion: u64` field. The manager picks `new_eversion =
       ex.eversion + 1` and sends it to the coordinator; the
       coordinator forwards it in every WriteShardReq it dispatches
       to data + parity targets.
       - WriteShardReq header: 20 → 28 bytes.

    2. `write_shard_local` writes `entry.eversion = new_eversion`
       (alongside `len`, `sealed_length`, `avali`) and persists via
       `save_meta`. The manager's `apply_ec_conversion_done` now
       *assigns* the same `new_eversion` to etcd (was `+= 1`) so all
       parties agree.

    3. Client passes `ex.eversion` in every ReadBytesReq.
       `handle_read_bytes` returns the new `CODE_EVERSION_MISMATCH = 6`
       (instead of an Err `FailedPrecondition`) when the client's
       eversion is older than local. Client surfaces this as a private
       `EversionStale` anyhow sentinel. Top-level
       `read_bytes_from_extent` wraps the body in a 2-attempt loop
       that calls `invalidate_extent_cache` and refetches `ExtentInfo`
       once on `EversionStale`.
       `read_replicated_with_failover` and `ec_subrange_read` /
       `ec_read_full` fail-fast on `EversionStale` (every replica
       reports the same mismatch by construction).

Performance note (user-asked):
  - Server-side ReadBytes adds one u64 comparison after the existing
    `extent.eversion` atomic load. ~1 ns.
  - Client-side adds one branch on resp.code before the existing
    OK check. ~1 ns.
  - WriteShardReq header grew 8 bytes; payload itself is shard-sized
    (~GB), overhead is 8e-9.
  - Retry loop runs once in the happy path (cache fresh) — same as
    before. Only the post-EC stale-cache path retries, and that
    saves multi-second timeouts.
