  Commit 1 — `fix(ucx): drop UcxEp close-on-Drop (was a noisy no-op)`
  (already pushed as 76a6f51):

    crates/transport/src/ucx/endpoint.rs
      - Removed `impl Drop for UcxEp` entirely (was: zero out raw + call
        `ucp_ep_close_nbx` with `UCP_EP_CLOSE_FLAG_FORCE`). The FORCE
        call was a noisy no-op: UCX rejects FORCE on EPs created with
        the default `UCP_ERR_HANDLING_MODE_NONE`, returning
        `UCS_ERR_INVALID_PARAM (-5)` synchronously and *not* closing
        the EP. So the call achieved nothing functional — just
        produced one `tracing::warn!` per EP teardown.
      - Updated the module-level "EP lifetime" doc to spell out why we
        now skip the close call entirely, and which alternatives were
        ruled out:
          * FORCE close requires PEER mode (rejected with -5 otherwise).
          * Graceful FLUSH-mode close deadlocks the loopback test (both
            peers in the same compio runtime block each other's ack).
          * `UCP_ERR_HANDLING_MODE_PEER` (which would let FORCE succeed)
            enables UCX's RC-layer dead-peer detection — under high
            fan-out write load (16 client threads × 16 inflight) rc_mlx5
            RNR/timeout fires *mid-test* and tears down live EPs
            (verified empirically: write ops/s collapsed from 16 k → 0
            around the 8-second mark). Worse than a teardown warning.
        Skipping the call leaks the C-side EP in the worker context
        until `ucp_worker_destroy` (thread/process exit). For autumn-rs
        this is fine: autumn-rpc::pool and autumn-stream::conn_pool
        reuse one EP per (peer-address, thread) pair for the life of
        the process, so EPs do not accumulate at runtime.

  Phase 2 cliff characterization sweep (2026-04-29 post-Phase-1, all
  with no `ucp_ep_close_nbx` warnings — Phase 1 fix holds across the
  range):
    --threads 16 : write 104 k / p99 11 ms · read 970 k / p99 0.46 ms ✓
    --threads 32 : write  80 k / p99 10 ms · read 610 k / p99 1.16 ms (degrading)
    --threads 64 : write  14 k / p99 54 ms · read 105 k / p99 18 ms (cliff)
    --threads 256: write   ~0           · read   0          (hard fail)

  RDMA / OS resource ceilings ruled out as the cause:
    - `ulimit -n` 1 048 576 (~500× headroom)
    - `ibv_devinfo -d mlx5_1 -v`: max_qp=131 072, max_mr=16 777 216
      (workload uses ~2 048 QPs at 256t = 1.5 % of cap)
    - `/proc/<ps_pid>/status` during 32t test: 26 threads, 14 GB RSS,
      healthy.
  Conclusion: bottleneck is the per-partition single-threaded
  user-space UCX worker, not RDMA / OS resources. Cliff is around
  ~512–1 024 concurrent UCX ops per worker on this host's rc_mlx5.
