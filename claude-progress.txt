Completed:
  - Root cause (F136): pre-EC extents have parity=[] (handle_create_stream + handle_stream_alloc_extent in
    crates/manager/src/rpc_handlers.rs:374,881). dispatch_recovery_task computes occupied = replicates ++ parity,
    so it can pick the FUTURE parity node as a recovery target. If a recovery dispatches before EC conversion
    runs, the recovering node ends up holding parity bytes (commit_shard_local rename overwrites recovery's
    .dat) AND the manager's apply_recovery_done later replaces replicates[slot] with that node id — resulting
    in the same node id appearing in both replicates and parity. Reads of the duplicated data shard return
    parity bytes → GC fails with "trailing bytes did not form a complete record".
  - Fix layered three ways in crates/manager/src/recovery.rs:
      1. dispatch_recovery_task early-returns when extent_id ∈ ec_conversion_inflight
      2. ec_conversion_dispatch_loop skips extents present in recovery_tasks
      3. apply_recovery_done rejects the apply if task.node_id is already in extent_nodes(ex) at a different
         slot, removing the stale recovery_tasks entry from memory + etcd so future re-dispatch is unblocked
  - Tests: f126_apply_recovery_done_rejects_duplicate_target and
    f126_apply_recovery_done_succeeds_when_target_is_unique in crates/manager/src/lib.rs.
