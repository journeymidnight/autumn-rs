Task scope: F102 — cluster.sh per-process start/stop for recovery testing.
Add `start-node N` / `stop-node N` / `restart-node N` / `start-ps` /
`stop-ps` / `restart-ps` subcommands so the user can drive
manager-side recovery dispatch and PS failover without bouncing the
whole cluster, without re-typing flags / env vars on every cycle.

Design choice (after user picked option (a)):
  Snapshot REPLICAS + CLUSTER_MODE + every AUTUMN_* env var to
  $DATA_ROOT/cluster_config at the end of `do_start`. Per-process
  subcommands `source` it back, so `stop-node 2 && start-node 2`
  reproduces the original launch line — including --3disk paths,
  --shards / --shard-stride, AUTUMN_BIND_HOST, etc.

Change set (cluster.sh only — no Rust source touched this session):
  - 6 new helpers introduced above do_start: compute_shard_config /
    launch_extent_node / register_extent_node / launch_ps /
    save_cluster_config / load_cluster_config.
  - do_start refactored to call those helpers (behavior-preserving;
    dropped the inline shards/loop + register-loop + PS launch block).
  - 4 new subcommand functions: do_start_node / do_stop_node /
    do_start_ps / do_stop_ps with guard rails (already-running
    detection, REPLICAS bounds check, missing-N usage error).
  - Main dispatcher: 6 new cases (start-node / stop-node / restart-node
    / start-ps / stop-ps / restart-ps). The integer positional parser
    grew an `ARG_INT_PROVIDED` flag so per-process subcommands can
    require an explicit N instead of silently defaulting to 1.
  - Mode validation (--3disk vs replicas==3, --multidisk-1node vs
    replicas==1) is now only enforced for start/restart/reset; the
    per-process subcommands inherit MODE from the snapshot.
  - Help text rewritten with two sections (Cluster / Per-process).

Known follow-up — F102-followup / F099-K-fix:
  `partition_server.rs` binary calls
  `PartitionServer::connect_with_advertise(...)` then `ps.serve(addr)`.
  `connect_with_advertise` → `finish_connect` → `sync_regions_once`
  runs BEFORE `serve()` calls `bind_listen_addr`, so on a restart
  (where manager etcd already knows about partitions), open_partition
  fires with `base_port == 0` and binds the per-partition listener to
  `0.0.0.0:1` instead of `:9201`. The fix is a one-liner —
  the `connect_with_advertise_and_port` variant already exists for
  exactly this case (lib.rs:661-670, calls bind_listen_addr BEFORE
  finish_connect):

    let ps = PartitionServer::connect_with_advertise_and_port(
        args.psid, &args.manager, Some(advertise), addr,
    ).await?;

  This bug also makes `cluster.sh restart` fail at the
  "wait_port 9201 partition 0 listener" step, not just `start-ps`.
  Diagnosed earlier this session via `lsof -p <ps>` showing
  `TCP *:1 (LISTEN)`.
