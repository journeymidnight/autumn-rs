TaskStatus: completed
Task scope: F117 — move all Reed-Solomon EC work (encode/decode/reconstruct) off the compio event loop onto compio::runtime::spawn_blocking, so extent_node + StreamClient stay responsive while EC math runs.

Current summary:
  Symptom: during `ec_conversion_dispatch_loop`-driven EC conversion,
  `autumn-extent` becomes sluggish on append/read RPCs even though no
  I/O is contended. Root cause: three CPU-bound Reed-Solomon call
  sites ran synchronously on a compio task, monopolising the event
  loop for 100–300 ms per 128 MiB sealed extent.

  Hazard sites scanned + fixed:

    1. crates/stream/src/extent_node.rs::handle_convert_to_ec @ 2849
       — `crate::erasure::ec_encode(&data, …)` was running on the
       extent-node compio runtime that also serves heartbeats +
       append + read RPCs. This is the user-reported "extent_node
       响应慢" hazard.

    2. crates/stream/src/extent_node.rs::run_ec_recovery_payload @ 2031
       — `crate::erasure::ec_reconstruct_shard(…)` rebuilds a single
       missing shard during recovery. The surrounding task is
       launched via `compio::runtime::spawn` (line 2493) which keeps
       it on the same compio runtime — *not* a blocking pool.

    3. crates/stream/src/client.rs::ec_read_full @ 1847 —
       `crate::erasure::ec_decode(shard_data, …)` runs on whichever
       caller's compio thread awaited the EC fallback path
       (typically PS P-log / P-bulk for row_stream / meta_stream
       full reads).

  Fix (one pattern, three sites):
    Wrap each call in `compio::runtime::spawn_blocking(move || { … })`,
    matching the partition-server SSTable-build pattern at
    crates/partition-server/src/lib.rs:2704. Each closure moves the
    input buffer (Vec<u8> payload for encode; Vec<Option<Vec<u8>>>
    shard set for decode/reconstruct) and Copy-typed shard counts.
    Error plumbing uses double `.map_err`+`?` to handle (i) the
    join-time panic-Box from compio's `JoinHandle<T> = Task<Result<T,
    Box<dyn Any + Send>>>` and (ii) the inner anyhow / (StatusCode,
    String) error from the erasure function.

  Hazards explicitly considered and ruled out of this fix:
    - WAL CRC32C in wal.rs:172/271 — bounded at ≤ 2 MiB per call
      (≈ 1 ms on hardware CRC), amortised by `write_batch`. Revisit
      only if profiling shows it on the hot path.
    - SSTable build (`build_sst_bytes`) — already on `spawn_blocking`
      since F088 (partition-server lib.rs:2704, :2955).
    - Bloom filter probe in handle_get — single key, O(µs); fine on
      the event loop.
    - WAL replay at startup — runs in `ExtentNode::new` before the
      event loop; not an issue.

Performance note:
  - spawn_blocking has a one-time cost of dispatching to a worker
    thread (~5–20 µs). Trivial vs the 100–300 ms RS encode/decode
    on a 128 MiB extent (~10⁴× cheaper). For the recovery path the
    surrounding compio::spawn task is also amortised over the 10s
    backoff retry loop so the dispatch cost is invisible.
