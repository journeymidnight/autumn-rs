Investigation:
  ✓ Re-read crates/transport/src/ucx/endpoint.rs send/recv path. Each
    call takes an OWNED `B: IoBuf` per send (no buffer reuse); UCX
    handles MR caching internally via UCX_MEMTYPE_CACHE (default `try`).
    Hypotheses #1 (buffer reuse) and #3 (MR re-registration) from
    F101-c were misdirected — neither is in our codepath.
  ✓ Triggered the wedge with --threads 1 --size 8M on a UCX cluster
    that had only `^sysv`. Single op took 11.5 s. PS log captured:
        mm_posix.c:233 UCX ERROR open(file_name=/proc/382025/fd/393
                                       flags=0x0) failed: No such file
                                       or directory
  ✓ Same family of failure as sysv: UCX `posix` transport reads peer
    process memory via /proc/<peer_pid>/fd/<N>. This environment
    blocks /proc fd visibility across processes. UCX picks posix for
    > eager rendezvous, open() fails, send wedges.

Fix:
  ✓ perf_check.sh: UCX_TLS default flipped from `^sysv` to
    `^sysv,^posix` (caller still overridable via `: "${UCX_TLS:=...}"`).
    UCX now falls back to `cma` (Cross-Memory-Attach, syscall-based,
    immune to IPC-namespace and /proc restrictions; ~17 GB/s in
    ucx_perftest) for intra-host bulk + `tcp` for control.
