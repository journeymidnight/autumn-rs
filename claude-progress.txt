Task scope: Fix split regression `etcdserver: duplicate key given in txn request`
  triggered by `split <part_id>` after the vp-table-refs change.

Current summary:
  Root cause: `handle_multi_modify_split` built one etcd txn containing
  `extents/<id>` writes from two sources:
  1. stream duplication (`compute_duplicate_stream`) updating `refs/eversion`
  2. right-child VP snapshot preview updating `vp_table_refs`

  When an extent appeared in both sets, the txn contained duplicate puts for
  the same etcd key, and etcd rejected it with `duplicate key given in txn request`.
  The two payloads also differed, so naive dedupe would have lost either the
  `refs` update or the `vp_table_refs` update.

  Fix implemented:
  1. added `AutumnManager::merge_extent_updates(base, overlays)`
  2. split path now merges overlapping extent updates by `extent_id`
  3. on overlap, preserve the stream-duplication mutation (`refs/eversion/...`)
     and overlay the new `vp_table_refs`
  4. added regression test `merge_extent_updates_preserves_ref_and_vp_changes`
